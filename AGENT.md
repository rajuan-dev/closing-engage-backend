# Backend Agent Rules

This file defines the required workflow for any agent or contributor creating or updating backend APIs in `closing-engage-backend`.

## Core Rule

Whenever you create a new API, you must also create its Swagger documentation in the same task.

Do not treat Swagger as optional or as a follow-up task.
API implementation is only considered complete when the Swagger documentation exists and is registered.

## Required Workflow For New APIs

When creating a new backend API:

1. Use the scaffold command first:
   - `npm run create:api`
   - or non-interactively:
   - `npm run create:api -- --name <module-name> --route <route-path> --tag "<Swagger Tag>" --entity <entity-name> --auth <none|admin|any|company|notary>`

2. The scaffold must create all of the following together:
   - route file
   - controller file
   - service file
   - Swagger documentation fragment
   - route registration
   - OpenAPI registration

3. After implementing the real endpoint logic, update the generated Swagger docs so they match:
   - request body
   - query params
   - path params
   - auth requirements
   - response schema
   - status codes
   - error cases

4. Before finishing, verify the backend still passes:
   - `npm run typecheck`

## Required Workflow For Existing APIs

Whenever an existing API is changed, its Swagger documentation must be updated in the same task if any of these changed:

- endpoint path
- HTTP method
- request payload
- response payload
- validation rules
- authentication or authorization
- status codes
- error behavior

## Completion Standard

A backend API task is not complete unless:

- the API code works
- the route is registered
- Swagger documentation exists
- Swagger documentation reflects the final API behavior
- typecheck passes

## Project Convention

Swagger fragments generated by the scaffold are automatically merged into:

- `src/docs/openapi.ts`

Generated module doc registrations are tracked in:

- `src/docs/generated-modules.ts`

## Agent Instruction

If the request includes creating an API, always end by ensuring Swagger API documentation has also been created or updated.
